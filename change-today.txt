# Application Architecture

## Overview

This application follows a layered architecture pattern with clear separation of concerns. Each layer has a specific responsibility and communicates only with adjacent layers.

## Architecture Flow

```
Actions → Services → Provider & Context → Hooks → Pages → PageSections → Components
```

## Layer Responsibilities

### 1. Actions Layer (`/src/actions`)

**Purpose**: Handle all API communication

**Responsibilities**:
- Make HTTP requests (GET, POST, PUT, PATCH, DELETE)
- Handle API endpoints and URLs
- Return raw API responses
- No business logic
- No state management

**Example**:
```typescript
// src/actions/user.actions.ts
class UserActions extends BaseAction {
  async fetchAll(): Promise<ApiResponseI<UserI[]>> {
    return await this.get(`${API_PATHS.AUTH}${this.endpoint}`);
  }
}
```

### 2. Services Layer (`/src/services`)

**Purpose**: Business logic and orchestration

**Responsibilities**:
- Call Actions to fetch/send data
- Transform data if needed
- Implement business rules
- Orchestrate multiple actions
- Validate data before sending
- Can aggregate data from multiple actions

**Example**:
```typescript
// src/services/users.service.ts
class UserService {
  async findAll(setUsers: (response: ApiResponseI<UserI[]>) => void) {
    const response = await userActions.fetchAll();
    // You could add business logic here (filtering, transformation, etc.)
    setUsers(response);
    return response;
  }
}
```

### 3. Provider & Context Layer (`/src/providers` & `/src/contexts`)

**Purpose**: State management and data distribution

**Responsibilities**:
- Manage application/feature state
- Provide state to child components
- Call Services to fetch/update data
- Handle loading states
- Distribute data through React Context

**Example**:
```typescript
// src/providers/users.provider.tsx
export const UsersProvider: React.FC<{ children: React.ReactNode }> = ({ children }) => {
  const [users, setUsers] = useState<ApiResponseI<UserI[]>>({ data: [] });
  const [isLoading, setIsLoading] = useState(false);

  const findAllUsers = async () => {
    setIsLoading(true);
    await userService.findAll(setUsers);
    setIsLoading(false);
  };

  return (
    <UsersContext.Provider value={{ users, isLoading, findAllUsers }}>
      {children}
    </UsersContext.Provider>
  );
};
```

### 4. Hooks Layer (`/src/hooks`)

**Purpose**: Reusable logic and context consumption

**Responsibilities**:
- Consume Context data
- Provide easy access to state and methods
- Can add component-specific logic
- Handle side effects (useEffect)

**Example**:
```typescript
// src/hooks/useUsers.ts
export const useUsers = () => {
  const context = React.useContext(UsersContext);
  if (context === undefined) {
    throw new Error("useUsers must be used within a UsersProvider");
  }
  return context;
};
```

### 5. Pages Layer (`/src/app/(Pages)`)

**Purpose**: Route-level components

**Responsibilities**:
- Define application routes
- Use hooks to get data
- Pass data to PageSections
- Handle page-level layout
- SEO and metadata

**Example**:
```typescript
// src/app/(Pages)/users/page.tsx
export default function UsersPage() {
  const { users, isLoading, findAllUsers } = useUsers();

  useEffect(() => {
    findAllUsers();
  }, []);

  return <UsersPageSection users={users} isLoading={isLoading} />;
}
```

### 6. PageSections Layer (`/src/app/(PageSections)`)

**Purpose**: Major sections of a page

**Responsibilities**:
- Organize page content into logical sections
- Receive data from Pages
- Pass data to Components
- Handle section-specific layout

**Example**:
```typescript
// src/app/(PageSections)/Users/UsersListSection.tsx
export function UsersListSection({ users, isLoading }) {
  return (
    <div>
      <h1>Users</h1>
      <UserTable users={users} loading={isLoading} />
    </div>
  );
}
```

### 7. Components Layer (`/src/components`)

**Purpose**: Reusable UI components

**Responsibilities**:
- Render UI elements
- Receive props from PageSections
- Handle user interactions
- Emit events to parent components
- No direct API calls
- No business logic

**Example**:
```typescript
// src/components/UserTable.tsx
export function UserTable({ users, loading }) {
  if (loading) return <Skeleton />;

  return (
    <table>
      {users.map(user => (
        <UserRow key={user.id} user={user} />
      ))}
    </table>
  );
}
```

## Data Flow Example

Let's trace a complete user creation flow:

```typescript
// 1. Component triggers action
<Button onClick={() => handleCreateUser(formData)} />

// 2. PageSection calls hook
function UsersPageSection() {
  const { createUser } = useUsers();

  const handleCreateUser = async (data) => {
    await createUser(data);
  };
}

// 3. Hook uses Context
export const useUsers = () => useContext(UsersContext);

// 4. Provider calls Service
const UsersProvider = () => {
  const createUser = async (data: CreateUserDataI) => {
    setCreateLoading(true);
    await userService.createUser(data);
    setCreateLoading(false);
    findAllUsers(); // Refresh list
  };
};

// 5. Service calls Action (with business logic if needed)
class UserService {
  async createUser(userData: CreateUserDataI) {
    // Could add validation, transformation here
    return await userActions.create(userData);
  }
}

// 6. Action makes API call
class UserActions extends BaseAction {
  async create(userData: CreateUserDataI) {
    return await this.post(
      `${API_PATHS.AUTH}${this.endpoint}`,
      userData,
      "User created successfully!"
    );
  }
}
```

## Benefits of This Architecture

1. **Separation of Concerns**: Each layer has a single, well-defined purpose
2. **Testability**: Each layer can be tested independently
3. **Maintainability**: Changes in one layer don't affect others
4. **Reusability**: Services and Actions can be used by multiple providers
5. **Scalability**: Easy to add new features following the same pattern
6. **Type Safety**: TypeScript interfaces flow through all layers

## Creating a New Feature

To add a new feature (e.g., "Products"), follow these steps:

1. **Create Action** (`src/actions/product.actions.ts`)
   ```typescript
   class ProductActions extends BaseAction {
     async fetchAll() { /* ... */ }
   }
   export const productActions = new ProductActions();
   ```

2. **Create Service** (`src/services/products.service.ts`)
   ```typescript
   class ProductService {
     async findAll(setProducts) {
       const response = await productActions.fetchAll();
       setProducts(response);
     }
   }
   export const productService = new ProductService();
   ```

3. **Create Context** (`src/contexts/products.context.tsx`)
   ```typescript
   export const ProductsContext = createContext<ProductsProps>({});
   ```

4. **Create Provider** (`src/providers/products.provider.tsx`)
   ```typescript
   export const ProductsProvider = ({ children }) => {
     const [products, setProducts] = useState([]);
     // ... implementation
   };
   ```

5. **Create Hook** (`src/hooks/useProducts.ts`)
   ```typescript
   export const useProducts = () => useContext(ProductsContext);
   ```

6. **Use in Page** (`src/app/(Pages)/products/page.tsx`)
   ```typescript
   export default function ProductsPage() {
     const { products } = useProducts();
     return <ProductsPageSection products={products} />;
   }
   ```

## Best Practices

- **Actions**: Keep them thin, only API calls
- **Services**: Add business logic, validation, data transformation
- **Providers**: Manage state, handle loading/error states
- **Hooks**: Keep simple, just context consumption
- **Pages**: Minimal logic, mostly routing and layout
- **PageSections**: Organize and structure page content
- **Components**: Pure presentation, no business logic

## Anti-Patterns to Avoid

❌ Making API calls directly in components
❌ Business logic in Actions
❌ State management in Services
❌ Direct API calls in Providers (should go through Services)
❌ Complex logic in Hooks (should be in Services)
